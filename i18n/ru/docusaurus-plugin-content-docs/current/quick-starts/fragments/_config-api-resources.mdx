Как только вы настроили ресурсы API, вы можете добавить их при настройке Logto в вашем приложении:

{props.configResourcesCode}

Каждый ресурс API имеет свои собственные разрешения (области действия).

Например, ресурс `https://shopping.your-app.com/api` имеет разрешения `shopping:read` и `shopping:write`, а ресурс `https://store.your-app.com/api` имеет разрешения `store:read` и `store:write`.

Чтобы запросить эти разрешения, вы можете добавить их при настройке Logto в вашем приложении:

{props.configResourcesWithScopesCode}

Вы можете заметить, что области действия определяются отдельно от ресурсов API. Это связано с тем, что [Resource Indicators for OAuth 2.0](https://www.rfc-editor.org/rfc/rfc8707.html) указывает, что конечные области действия для запроса будут декартовым произведением всех областей действия на всех целевых сервисах.

{props.configResourcesWithSharedScopesCode &&

<>

<p>
  Таким образом, в приведенном выше случае области действия могут быть упрощены из определения в
  Logto, оба ресурса API могут иметь области действия <code>read</code> и <code>write</code> без
  префикса. Затем, в конфигурации Logto:
</p>

{props.configResourcesWithSharedScopesCode}

<p>
  Для каждого ресурса API будет запрашиваться как <code>read</code>, так и <code>write</code>{' '}
  области действия.
</p>

</>
}

:::note
Допустимо запрашивать области действия, которые не определены в ресурсах API. Например, вы можете запросить область действия `email`, даже если в ресурсах API нет доступной области действия `email`. Недоступные области действия будут безопасно игнорироваться.
:::

После успешного входа Logto выдаст соответствующие области действия ресурсам API в соответствии с ролями пользователя.
