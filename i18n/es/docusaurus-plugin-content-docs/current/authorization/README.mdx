# Autorización (Authorization)

[Autorización (Authorization)](https://auth.wiki/authorization) en Logto define **qué pueden hacer los usuarios y aplicaciones después de la autenticación**: qué APIs, recursos o acciones están permitidas para cada identidad.

Logto proporciona una autorización flexible basada en tokens para aplicaciones SaaS y de IA modernas. Puedes proteger recursos de API a nivel global o dentro del contexto de cada organización. Todos los permisos se gestionan a través de un sistema de [control de acceso basado en roles (RBAC)](/authorization/role-based-access-control), con soporte avanzado para aplicaciones multi-tenant mediante [plantillas de organización](/authorization/organization-template). Más allá de controlar lo que los usuarios pueden hacer, también puedes restringir qué usuarios pueden iniciar sesión en cada aplicación con el [control de acceso a nivel de aplicación](/authorization/app-level-access-control).

## Conceptos clave \{#core-concepts}

- **Control de acceso basado en roles (RBAC):** Logto utiliza RBAC como base para asignar permisos a usuarios, clientes y servicios. [Aprende más sobre RBAC](/authorization/role-based-access-control).
- **Recurso de API (API resource):** Cualquier servicio backend o endpoint que quieras proteger (global o específico de la organización).
- **Rol (Role):** Un grupo de permisos (por ejemplo, admin, visualizador, editor).
- **Permiso (scope):** Una acción permitida específica (por ejemplo, `read:report`, `invite:member`).
- **Organización (Organization):** Representa un tenant, espacio de trabajo o cliente en tu aplicación. **Esto es diferente del tenant de Logto, que se refiere a tu proyecto o instancia general de Logto**.
- **Plantilla de organización (Organization template):** Para aplicaciones multi-tenant, define un conjunto reutilizable de roles y permisos aplicados en todas las organizaciones. [Mira cómo funcionan las plantillas de organización](/authorization/organization-template).
- **Token de acceso / token de organización (Access token / organization token):** Tokens que contienen reclamos para permisos globales o con alcance de organización.

## Escenarios de autorización \{#authorization-scenarios}

Hay tres patrones principales de autorización en Logto. Elige el escenario que se adapte a tus necesidades:

| Escenario                                               | Cuándo usar                                                                                               | Tipo de token         | Configuración de roles                                              | Más información                                                                                     |
| ------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- | --------------------- | ------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **Permisos globales de recursos de API**                | Proteger recursos de API compartidos en todo tu tenant de Logto (no específicos de organización)          | Token de acceso       | Asignar roles/permisos globales                                     | [Proteger recursos de API globales](/authorization/global-api-resources)                            |
| **Permisos de organización (no API)**                   | Controlar acciones específicas de la organización, características de la UI o lógica de negocio (no APIs) | Token de organización | Asignar roles/permisos de organización para controles de la app     | [Proteger permisos de organización (no API)](/authorization/organization-permissions)               |
| **Permisos de recursos de API a nivel de organización** | Proteger recursos de API accesibles dentro de una organización específica                                 | Token de organización | Asignar roles/permisos de organización para APIs de la organización | [Proteger recursos de API a nivel de organización](/authorization/organization-level-api-resources) |

Logto modela los recursos de API según [RFC 8707](https://auth.wiki/resource-indicator), usando el parámetro `resource` en los flujos de autorización OAuth 2.0. Esto facilita asegurar múltiples APIs o microservicios y garantiza la compatibilidad con otros sistemas basados en estándares.

Estos patrones controlan lo que un usuario puede hacer **después** de iniciar sesión en una aplicación. Para controlar si un usuario puede iniciar sesión en una aplicación específica, utiliza el [control de acceso a nivel de aplicación](/authorization/app-level-access-control). Restringe el acceso a la aplicación con reglas de permiso basadas en usuarios, roles, organizaciones o roles de organización, y funciona de manera independiente de los patrones basados en tokens anteriores.

:::tip
¿Necesitas reclamos personalizados o control de acceso avanzado? Consulta [Reclamos personalizados en tokens](/developers/custom-token-claims).
:::

## Cómo funciona la autorización en Logto \{#how-logto-authorization-works}

- **Basado en tokens:** Cada acceso se concede mediante un token de acceso seguro y firmado. Tu backend valida el token y aplica los permisos (alcances).
- **Permiso global vs. de organización (scope):**

  - Permiso _global_ (scope): Controla el acceso a recursos de API en todo tu tenant de Logto.
  - Permiso _de organización_ (scope): Controla tanto la lógica de negocio (características de la app) como los recursos de API dentro del contexto de una organización. Los permisos de organización pueden aplicarse a características que no son API (como elementos de UI o flujos de trabajo) y / o endpoints de API con alcance de organización.

- **Roles y permisos (alcances):** Los roles son colecciones de permisos (alcances). Asigna roles a usuarios o clientes a nivel global o dentro de una organización, según tu escenario.
