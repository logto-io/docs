# ทดสอบประสบการณ์ผู้ใช้ด้วย Live preview

คุณสามารถใช้ฟีเจอร์ **Live preview** เพื่อทดสอบการตั้งค่าประสบการณ์การลงชื่อเข้าใช้ของคุณได้อย่างสะดวก

## Live preview \{#live-preview}

ไปที่ <CloudLink to="/sign-in-experience/sign-up-and-sign-in">Console > Sign-in & account > Sign-up and sign-in</CloudLink> แล้วคลิกปุ่ม "Live preview" ที่มุมขวาบน จะเปิดแท็บเบราว์เซอร์ใหม่พร้อมเดโมที่ใช้งานได้เต็มรูปแบบของความสามารถของ Logto เดโมนี้จะแสดงแนวทางการออกแบบประสบการณ์สำหรับขั้นตอนการลงชื่อเข้าใช้และลงทะเบียน ซึ่งสามารถปรับแต่งได้สูงและผสานรวมเข้ากับแอปของคุณได้อย่างไร้รอยต่อ

:::note

ในตัวอย่างนี้ คุณควรสร้างบัญชีก่อนเพื่อทดสอบประสบการณ์ดังกล่าว คุณสามารถจัดการบัญชีนี้ได้ภายหลังที่ <CloudLink to="/user-management">Console > User management</CloudLink>

:::

หากคุณลองลงชื่อเข้าใช้ด้วยบัญชีผู้ใช้ที่สร้างขึ้นและกระบวนการสำเร็จโดยไม่มีปัญหา คุณจะถูกเปลี่ยนเส้นทางไปยังหน้าสำเร็จที่แสดงตัวระบุและ ID ของคุณ

## Dev panel \{#dev-panel}

หลังจากลงชื่อเข้าใช้สำเร็จด้วย Live Preview ของ Logto คุณจะเข้าสู่แดชบอร์ดที่มีสองการกระทำหลัก:

- **ปุ่ม Sign out the live preview**: ยุติเซสชันปัจจุบัน
- **ปุ่ม Open dev panel**: เปิดเครื่องมือดีบักสำหรับวิเคราะห์ flow การยืนยันตัวตนของคุณ

Development Panel ให้ข้อมูลเชิงลึกแบบเรียลไทม์สำหรับการแก้ไขปัญหาการเชื่อมต่อ OIDC การตรวจสอบโทเค็นและการอ้างสิทธิ์ (claims) และการทดสอบสถานการณ์เอกลักษณ์ขั้นสูง

### Logto Config \{#logto-config}

Live Preview อาศัยแอปเดโมที่ซ่อนอยู่ซึ่งมีประสบการณ์การลงชื่อเข้าใช้แบบรวมศูนย์เป็นค่าเริ่มต้น ในส่วน **Logto Config** คุณสามารถกำหนดค่าพารามิเตอร์การยืนยันตัวตนที่รวมอยู่ในคำขอการยืนยันตัวตน เพื่อสั่งให้ Logto OIDC endpoint จัดการกระบวนการยืนยันตัวตน เช่น การกำหนด app ID, การระบุ organization ID, การร้องขอขอบเขต (scopes) เฉพาะ ฯลฯ การตั้งค่านี้คล้ายกับการเริ่มต้น LogtoClient ด้วย `LogtoProvider` ซึ่งสร้างบริบท Logto ให้กับแอปของคุณ

:::note

อย่าลืมบันทึกการเปลี่ยนแปลงการตั้งค่าของคุณ การเปลี่ยนแปลงจะมีผลในครั้งถัดไปที่คุณลงชื่อเข้าใช้ด้วย Live Preview

:::

- **App ID**: ระบุ App ID เฉพาะของคุณ มีความสำคัญสำหรับการทดสอบ [ประสบการณ์การลงชื่อเข้าใช้เฉพาะแอป](/customization/match-your-brand#app-specific-branding) และนโยบายการปกป้องทรัพยากร
- **Sign-in extra parameters**: ส่ง [พารามิเตอร์การยืนยันตัวตน](/end-user-flows/authentication-parameters) เพิ่มเติมเพื่อทดสอบกระบวนการลงชื่อเข้าใช้แบบกำหนดเอง ตัวอย่างเช่น:
  - `organization_id=xyz123456`: ทดสอบ [ประสบการณ์การลงชื่อเข้าใช้เฉพาะองค์กร](/customization/match-your-brand#organization-specific-branding) หรือ [สิทธิ์ขององค์กร](/authorization/organization-permissions)
  - `direct_sign_in=social:google`: [เรียกใช้การลงชื่อเข้าใช้โซเชียลโดยตรง](/end-user-flows/authentication-parameters/direct-sign-in#social-sign-in) ด้วยตัวเชื่อมต่อ Google
  - `direct_sign_in=sso:1234567890`: [เรียกใช้ enterprise SSO โดยตรง](/end-user-flows/authentication-parameters/direct-sign-in#enterprise-sso) ด้วยตัวเชื่อมต่อองค์กรที่ระบุ
  - `ui_locales=fr-CA fr en`: บังคับให้ UI การลงชื่อเข้าใช้ใช้ภาษาตามแท็กแรกที่รองรับ ส่งผลต่อการแปลอีเมลและเปิดเผย `uiLocales` ในเทมเพลต
- **Prompt**: เพิ่ม [ค่า OIDC prompt](https://blog.logto.io/introducing-oidc-prompt) (คั่นด้วยช่องว่าง) เพื่อระบุว่า Authorization Server จะขอให้ผู้ใช้ปลายทางยืนยันตัวตนใหม่หรือขอความยินยอมหรือไม่ ค่าที่กำหนดได้แก่:
  - `login`: บังคับให้ผู้ใช้ยืนยันตัวตนใหม่
  - `consent`: ต้องการความยินยอมจากผู้ใช้ก่อนแบ่งปันข้อมูลกับ client
  - `none`: ระบุว่าไม่ควรแสดง UI ใด ๆ ใช้สำหรับตรวจสอบการยืนยันตัวตนหรือความยินยอมที่มีอยู่แล้ว
- **Scope**: ระบุขอบเขต (scopes; คั่นด้วยช่องว่าง) สำหรับการร้องขอการอ้างสิทธิ์ (claims) ตัวอย่างเช่น:
  - หากต้องการขอรายละเอียดองค์กรของผู้ใช้ ให้ใช้สองขอบเขต: `urn:logto:scope:organizations urn:logto:scope:organization_roles`
  - หากต้องการขอสิทธิ์ API ให้ระบุ **ชื่อ scope** ในช่อง **Scope** และกรอก **API identifier** ที่เกี่ยวข้องในช่อง **Resource** ด้านล่าง
- **Resource**: กรอกตัวระบุทรัพยากร API (คั่นด้วยช่องว่าง) ที่แอปของคุณต้องการเข้าถึง ช่องนี้ช่วยให้แอปของคุณร้องขอ access token สำหรับทรัพยากรเฉพาะตามที่ต้องการ

### Refresh token grant \{#refresh-token-grant}

ส่วนนี้อธิบายวิธี [รีเฟรชโทเค็น](https://auth-wiki.logto.io/refresh-token) สำหรับประเภทการเข้าถึงต่าง ๆ

- **Resource**: กรอก API Identifier เฉพาะ (คั่นด้วยช่องว่าง) หากคุณต้องการรีเฟรช access token สำหรับทรัพยากร API
- **Organization ID**: กรอก organization ID (คั่นด้วยช่องว่าง) หากคุณต้องการรีเฟรชโทเค็นองค์กร

ดู [การอนุญาต (Authorization)](/authorization) สำหรับรายละเอียดเพิ่มเติมเกี่ยวกับการใช้ refresh token grant

:::note
ใส่ `consent` ในพารามิเตอร์ prompt สำหรับ `offline_access` จะมีการออก `refresh token` ก็ต่อเมื่อ prompt มี `consent`
:::

### User info \{#user-info}

ตรวจสอบข้อมูลผู้ใช้ที่ได้จาก flow การยืนยันตัวตนของคุณ:

- **Get ID token claims**: ดึงการอ้างสิทธิ์ (claims) ทั้งหมดที่อยู่ใน [ID token](https://auth-wiki.logto.io/id-token) เช่น `at_hash`, `aud`, `auth_time`, `exp`, `iat`, `issu` และโปรไฟล์ผู้ใช้เพิ่มเติม ใช้สำหรับตรวจสอบความถูกต้องของโทเค็นและดึงข้อมูลผู้ใช้ที่จำเป็น
- **Fetch user info**: ดึงเฉพาะข้อมูลโปรไฟล์ผู้ใช้โดยการเรียก userinfo endpoint

:::note

ใช้ Developer Tools ของ Chrome (F12 → Console) เพื่อตรวจสอบ network requests และรีวิวโทเค็นกับการอ้างสิทธิ์ (claims)

:::
