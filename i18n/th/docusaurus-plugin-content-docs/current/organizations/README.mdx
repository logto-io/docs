# องค์กร (Organizations)

ฟีเจอร์องค์กรมีประสิทธิภาพอย่างยิ่งในซอฟต์แวร์ SaaS แบบหลายผู้เช่า (multi-tenant) และ [แอปพลิเคชันแบบธุรกิจต่อธุรกิจ (B2B)](/introduction/plan-your-architecture/b2b) นอกจากผู้ใช้แต่ละรายแล้ว ลูกค้าของคุณยังอาจประกอบด้วยทีม องค์กร หรือทั้งบริษัท

ด้วยองค์ประกอบพื้นฐานนี้ คุณสามารถสร้างฟีเจอร์สำคัญสำหรับ [แอปพลิเคชันแบบหลายผู้เช่า (multi-tenancy)](https://auth.wiki/multi-tenancy) เช่น:

- ผลิตภัณฑ์ที่สามารถใช้งานได้โดยหลายองค์กร
- การจัดเตรียมสมาชิกองค์กรผ่าน [การเชิญ (invitation)](/end-user-flows/organization-experience/invite-organization-members) หรือ [Just-in-Time](/organizations/just-in-time-provisioning)
- [การควบคุมการเข้าถึง (Access controls)](/authorization/organization-template) ที่กำหนดโดยบทบาทที่มอบหมายให้สมาชิกภายในองค์กร
- เชื่อมโยงองค์กรกับประสบการณ์ [Enterprise SSO (Single Sign-On)](/end-user-flows/enterprise-sso)

คำว่า "องค์กร" ยังถูกใช้ในรูปแบบอื่น เช่น "workspace", "team", "company" ฯลฯ ใน Logto เราใช้ "องค์กร" เป็นคำกลางเพื่อแสดงแนวคิดของ multi-tenancy

## ข้อควรพิจารณาในการวางแผน \{#planning-considerations}

เมื่อใช้งานองค์กร การผสานรวมอาจต้องใช้ความพยายามในแต่ละระดับที่แตกต่างกัน เราแนะนำให้คุณอ้างอิงข้ามหัวข้อเหล่านี้เพื่อปรับแต่งการพัฒนา

- **ประสบการณ์องค์กร**: ประสบการณ์ขององค์กรควรเป็นอย่างไร? ผู้ดูแลองค์กรสามารถสร้างองค์กรของตนเองได้หรือไม่? ผู้ดูแลองค์กรสามารถเชิญสมาชิกได้หรือไม่? คุณตั้งใจจะสร้างแดชบอร์ดแบบกำหนดเองที่อนุญาตให้ผู้ดูแลจัดการข้อมูลเกี่ยวกับตัวตนของสมาชิกองค์กรได้ด้วยตนเองหรือไม่?
- **บทบาทและการควบคุมการเข้าถึง**: แอปพลิเคชันจำเป็นต้องให้ผู้ใช้มีบทบาทเฉพาะภายในองค์กรของตนหรือไม่?
- **จัดการองค์กรผ่าน Logto Console และ Management API:** คุณต้องการตั้งค่าการกำหนดค่าใดโดยตรงใน Logto Console และ Management API?

สำหรับแหล่งข้อมูลการวางแผนเพิ่มเติม สำรวจสองหัวข้อต่อไปนี้เพื่อหาแนวคิดและแรงบันดาลใจ
