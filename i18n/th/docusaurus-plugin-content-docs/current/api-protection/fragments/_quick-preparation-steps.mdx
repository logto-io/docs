## ขั้นตอนเตรียมความพร้อมอย่างรวดเร็ว \{#quick-preparation-steps}

### กำหนดค่าทรัพยากรและสิทธิ์ของ Logto \{#configure-logto-resources-permissions}

<Tabs groupId="permission-models">
  <TabItem value="global-api-resources" label="ทรัพยากร API ระดับโกลบอล">

    1. **สร้างทรัพยากร API:** ไปที่ [Console → ทรัพยากร API](https://cloud.logto.io/api-resources) และลงทะเบียน API ของคุณ (เช่น `https://api.yourapp.com`)
    2. **กำหนดสิทธิ์:** เพิ่มขอบเขต (scopes) เช่น `read:products`, `write:orders` – ดู [กำหนดทรัพยากร API พร้อมสิทธิ์](/authorization/role-based-access-control#define-api-resources-with-permissions)
    3. **สร้างบทบาทระดับโกลบอล:** ไปที่ [Console → บทบาท](https://cloud.logto.io/roles) และสร้างบทบาทที่รวมสิทธิ์ API ของคุณ – ดู [กำหนดค่าบทบาทระดับโกลบอล](/authorization/role-based-access-control#configure-global-roles)
    4. **กำหนดบทบาท:** กำหนดบทบาทให้กับผู้ใช้หรือแอป M2M ที่ต้องการเข้าถึง API

  </TabItem>
  <TabItem value="organization-permissions" label="สิทธิ์ขององค์กร (ไม่ใช่ API)">

    1. **กำหนดสิทธิ์ขององค์กร:** สร้างสิทธิ์ขององค์กรที่ไม่ใช่ API เช่น `invite:member`, `manage:billing` ในเทมเพลตขององค์กร
    2. **ตั้งค่าบทบาทขององค์กร:** กำหนดค่าเทมเพลตขององค์กรด้วยบทบาทเฉพาะองค์กรและกำหนดสิทธิ์ให้กับบทบาทเหล่านั้น
    3. **กำหนดบทบาทขององค์กร:** กำหนดผู้ใช้ให้กับบทบาทขององค์กรในแต่ละบริบทขององค์กร

  </TabItem>
  <TabItem value="organization-level-api-resources" label="ทรัพยากร API ระดับองค์กร">

    1. **สร้างทรัพยากร API:** ลงทะเบียนทรัพยากร API ของคุณเช่นเดียวกับข้างต้น แต่จะใช้ในบริบทขององค์กร
    2. **กำหนดสิทธิ์:** เพิ่มขอบเขต (scopes) เช่น `read:data`, `write:settings` ที่จำกัดในบริบทขององค์กร
    3. **กำหนดค่าเทมเพลตขององค์กร:** ตั้งค่าบทบาทขององค์กรที่รวมสิทธิ์ของทรัพยากร API ของคุณ
    4. **กำหนดบทบาทขององค์กร:** กำหนดผู้ใช้หรือแอป M2M ให้กับบทบาทขององค์กรที่รวมสิทธิ์ API
    5. **ตั้งค่าหลายผู้เช่า:** ตรวจสอบให้แน่ใจว่า API ของคุณสามารถจัดการข้อมูลและการตรวจสอบที่จำกัดในแต่ละองค์กรได้

  </TabItem>
</Tabs>

:::tip ใหม่กับ RBAC?
เริ่มต้นด้วย [คู่มือการควบคุมการเข้าถึงตามบทบาท (RBAC)](/authorization/role-based-access-control) ของเรา สำหรับคำแนะนำการตั้งค่าแบบทีละขั้นตอน
:::

### อัปเดตแอปพลิเคชันฝั่งไคลเอนต์ของคุณ \{#update-your-client-application}

**ร้องขอขอบเขต (scopes) ที่เหมาะสมในไคลเอนต์ของคุณ:**

- การยืนยันตัวตนผู้ใช้: [อัปเดตแอปของคุณ →](/quick-starts) เพื่อร้องขอขอบเขต API และ/หรือบริบทขององค์กร
- เครื่องต่อเครื่อง: [กำหนดค่า M2M scopes →](/quick-starts/m2m) สำหรับการเข้าถึงระหว่างเซิร์ฟเวอร์

กระบวนการนี้มักเกี่ยวข้องกับการอัปเดตการกำหนดค่าไคลเอนต์ของคุณเพื่อรวมหนึ่งหรือมากกว่ารายการต่อไปนี้:

- พารามิเตอร์ `scope` ในกระบวนการ OAuth
- พารามิเตอร์ `resource` สำหรับการเข้าถึงทรัพยากร API
- `organization_id` สำหรับบริบทขององค์กร

:::tip ก่อนเริ่มเขียนโค้ด
ตรวจสอบให้แน่ใจว่าผู้ใช้หรือแอป M2M ที่คุณทดสอบได้รับการกำหนดบทบาทหรือบทบาทขององค์กรที่มีสิทธิ์ที่จำเป็นสำหรับ API ของคุณแล้ว
:::
