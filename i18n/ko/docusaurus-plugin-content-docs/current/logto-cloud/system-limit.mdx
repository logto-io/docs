# 시스템 한도

시스템 한도는 Logto의 안정성과 공정성을 보장합니다. 이는 과금과는 별개입니다.

**요약**

- **Dev 테넌트:** 고정 상한 (하드 리밋)
- **Pro 테넌트:** 시작 상한 (소프트 리밋). 정당한 사용의 경우 상향 조정이 가능합니다.
- **Enterprise 테넌트:** 계약서에 정의된 대로 구성 가능

가격 페이지에서 **무제한**은 *공정 사용 하에 사전 설정된 과금 상한이 없음*을 의미합니다. 플랫폼의 안정성을 보호하기 위해 **초기 가드레일**(엔티티 상한 및 속도 제한 등)이 적용되며, 정당한 워크로드에 따라 상향 조정될 수 있습니다.

## 한도의 종류 \{#types-of-limits}

### 시스템 리소스 및 엔티티 상한 \{#system-resource-and-entity-caps}

애플리케이션, 역할, 조직, 커넥터, Webhook 등과 같은 객체의 최대 개수 및 액세스 토큰 (Access token) 발급과 같은 일부 사용량 기반 시스템 리소스의 최대치.

### 속도 제한 (Rate limit) \{#rate-limit}

테넌트별로 적용되는 요청 수준 보호.

기본값: **테넌트당 10초에 200건의 요청** (토큰 버킷 방식). 이는 대략 **초당 20건의 지속 요청**에 해당하며, 버킷 용량 내에서 짧은 버스트가 허용됩니다.

트래픽이 정당하며 이 한도를 지속적으로 초과하는 경우, 예외 적용을 검토할 수 있습니다.

## 한도에 도달하면 어떻게 되나요? \{#what-happens-when-you-hit-a-limit}

가드레일이 작동하면 요청이 다음과 같은 오류로 실패할 수 있습니다:

- **HTTP 429** (속도 제한)
- **HTTP 4xx** 및 한도 관련 메시지 (엔티티 상한 또는 기능 가드레일)

이로 인해 데이터가 삭제되지는 않습니다. 한도를 초과한 작업만 차단됩니다.

## 한도 상향 요청 방법 \{#request-a-limit-increase}

한도에 도달했으며 사용 사례가 실제라면 저희에게 연락하세요. Pro 및 Enterprise의 경우, 일반적으로 한도를 신속하게 상향할 수 있습니다.

다음 정보를 포함해 주세요:

- 테넌트 ID (또는 테넌트 도메인)
- 초과한 한도 (기능명 + 필요한 수치)
- (RPS 요청의 경우) 예상 트래픽: 평균 RPS, 최대 RPS, 버스트 패턴
- 영향받는 엔드포인트 (예: 로그인, 토큰, JWKS, userinfo, Management API)
- 필요 시점 (언제 상향이 필요한지)

## 테넌트 수준 한도 보호 \{#tenant-level-rate-limit-protection}

### Dev 테넌트 \{#dev-tenants}

[Dev 테넌트](/logto-cloud/tenant-settings#development)는 개발 및 테스트 용도로 제공됩니다.

Dev 테넌트 한도는 **하드 리밋**으로, 개발 워크로드에 맞게 설정되어 있습니다. 프로덕션 환경이 필요하다면 [테넌트를 Pro로 전환](/logto-cloud/tenant-settings#tenant-types-dev-vs-prod)할 수 있습니다.

Dev 테넌트는 별도의 [데이터 보존 정책](/logto-cloud/dev-tenant-data-retention)도 따릅니다.
