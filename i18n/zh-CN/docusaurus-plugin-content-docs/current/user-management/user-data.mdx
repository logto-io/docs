# 用户数据结构

用户是身份服务中的核心实体。在 Logto 中，它们包括基于 OpenID Connect 协议的基本认证数据以及自定义数据。

## 用户资料

每个用户都有一个包含 [所有用户信息](./#property-reference) 的资料。

它由以下类型的数据组成：

- [基本数据](./#basic-data)：是用户资料中的基本信息。它存储所有其他 _用户_ 的属性，除了社交 `identities` 和 `custom_data`，例如用户 ID、用户名、电子邮件、电话号码以及用户上次登录的时间。
- [社交身份](./#social-identities)：存储从社交登录（即使用社交连接器登录）中检索到的用户信息，例如 Facebook、GitHub 和微信。
- [自定义数据](./#custom-data)：存储未列在预定义用户属性中的其他用户信息，例如用户偏好的颜色和语言。

以下是从 Facebook 登录中检索到的用户数据示例：

```json
{
  "id": "iHXPuSb9eMzt",
  "username": null,
  "primaryEmail": null,
  "primaryPhone": null,
  "name": "John Doe",
  "avatar": "https://example.com/avatar.png",
  "customData": {
    "preferences": {
      "language": "en",
      "color": "#f236c9"
    }
  },
  "identities": {
    "facebook": {
      "userId": "106077000000000",
      "details": {
        "id": "106077000000000",
        "name": "John Doe",
        "email": "johndoe@logto.io",
        "avatar": "https://example.com/avatar.png"
      }
    }
  },
  "lastSignInAt": 1655799453171,
  "applicationId": "admin_console"
}
```

你可以使用 <CloudLink to="/users">Logto Console</CloudLink> 或 Logto Management API 查询用户资料，例如 [`GET /api/users/:userId`](https://openapi.logto.io/operation/operation-getuser)。

## 基本数据

让我们逐步了解用户 _基本数据_ 中的所有属性。

### id

_id_ 是一个唯一的自动生成的键，用于在 Logto 中标识用户。

### username

_username_ 用于使用 _username_ 和密码登录。

其值来自用户首次注册时的用户名。它可能为 `null`。其非空值不应超过 128 个字符，只能包含字母、数字和下划线 (`_`)，且不能以数字开头。

### primary_email

_primary_email_ 是用户的电子邮件地址，用于使用电子邮件和验证码登录。

其值通常来自用户首次注册时的电子邮件地址。它可能为 `null`。其最大长度为 128。

### primary_phone

_primary_phone_ 是用户的电话号码，用于使用电话号码和来自短信的验证码登录。

其值通常来自用户首次注册时的电话号码。它可能为 `null`。其非空值应包含以 [国家区号](https://en.wikipedia.org/wiki/List_of_country_calling_codes) 为前缀的数字（不包括加号 `+`）。

### name

_name_ 是用户的全名。其最大长度为 128。

### avatar

_avatar_ 是指向用户头像图像的 URL。其最大长度为 2048。

如果用户使用 Google 和 Facebook 等社交连接器注册，其值可能会从社交用户信息中检索到。

:::note

此属性映射到 [OpenID Connect](https://openid.net/connect/) 标准中的 `picture` 声明。

:::

### profile

_profile_ 存储其他 OpenID Connect [标准声明](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims)，这些声明未包含在用户属性中。

其类型定义可以在 [此文件](https://github.com/logto-io/logto/blob/HEAD/packages/schemas/src/foundations/jsonb-types/users.ts#L6) 中找到。以下是类型定义的副本：

```tsx
type UserProfile = Partial<{
  familyName: string;
  givenName: string;
  middleName: string;
  nickname: string;
  preferredUsername: string;
  profile: string;
  website: string;
  gender: string;
  birthdate: string;
  zoneinfo: string;
  locale: string;
  address: Partial<{
    formatted: string;
    streetAddress: string;
    locality: string;
    region: string;
    postalCode: string;
    country: string;
  }>;
}>;
```

:::note

`Partial` 意味着所有属性都是可选的。

:::

与其他标准声明相比的一个区别是，`profile` 中的属性只有在其值不为空时才会包含在 ID 令牌或用户信息端点响应中，而其他标准声明在值为空时将返回 `null`。

### application_id

_application_id_ 的值来自用户首次登录的应用程序。它可能为 `null`。

### last_signed_in_at

_last_signed_in_at_ 是用户上次登录时的带时区的时间戳。

### password_encrypted

_password_encrypted_ 用于存储用户的加密密码。

其值来自用户首次注册时的密码。它可能为 `null`。如果其值为非空，则加密前的原始内容应至少为六个字符。

### password_encryption_method

_password_encryption_method_ 用于加密用户的密码。其值在用户使用用户名和密码注册时初始化。它可能为 `null`。

Logto 默认使用 [Argon2](https://en.wikipedia.org/wiki/Argon2) 的实现 [node-argon2](https://github.com/ranisalt/node-argon2) 作为加密方法；如果你感兴趣，请参阅参考资料以获取详细信息。

从密码为 `123456` 的用户中抽样一个 _password_encrypted_ 和 _password_encryption_method_：

```json
{
  "password_encryption_method": "Argon2i",
  "password_encrypted": "$argon2i$v=19$m=4096,t=10,p=1$aZzrqpSX45DOo+9uEW6XVw$O4MdirF0mtuWWWz68eyNAt2u1FzzV3m3g00oIxmEr0U"
}
```

### is_suspended

_is_suspended_ 是一个布尔值，指示用户是否被暂停。该值可以通过调用 [Logto Management API](https://openapi.logto.io/operation/operation-updateuserissuspended) 或使用管理控制台进行管理。

一旦用户被暂停，预先授予的刷新令牌将立即被撤销，用户将无法再通过 Logto 进行认证。

## 社交身份

_identities_ 包含从社交登录（即使用社交连接器登录）中检索到的用户信息。每个用户的 _identities_ 存储在一个单独的 JSON 对象中。

用户信息因社交身份提供商（即社交网络平台）而异，通常包括以下内容：

- 身份提供商的 _target_，例如 "facebook" 或 "google"
- 用户在此提供商的唯一标识符
- 用户的姓名
- 用户的已验证电子邮件
- 用户的头像

用户的帐户可以通过社交登录链接到多个社交身份提供商；从这些提供商检索到的相应用户信息将存储在 _identities_ 对象中。

从一个使用 Google 和 Facebook 登录的用户中抽样 _identities_：

```json
{
  "facebook": {
    "userId": "5110888888888888",
    "details": {
      "id": "5110888888888888",
      "name": "John Doe",
      "email": "johndoe@logto.io",
      "avatar": "https://example.com/avatar.png"
    }
  },
  "google": {
    "userId": "111000000000000000000",
    "details": {
      "id": "111000000000000000000",
      "name": "John Doe",
      "email": "johndoe@gmail.com",
      "avatar": "https://example.com/avatar.png"
    }
  }
}
```

## 自定义数据

_custom_data_ 存储未列在预定义用户属性中的其他用户信息。

你可以使用 _custom_data_ 做以下事情：

- 记录用户是否已完成特定操作，例如是否已查看欢迎页面。
- 在用户资料中存储应用程序特定的数据，例如用户每个应用程序的首选语言和外观。
- 维护与用户相关的其他任意数据。

从 Logto 中的管理员用户中抽样 _custom_data_：

```json
{
  "adminConsolePreferences": {
    "language": "en",
    "appearanceMode": "system",
    "experienceNoticeConfirmed": true
  },
  "customDataFoo": {
    "foo": "foo"
  },
  "customDataBar": {
    "bar": "bar"
  }
}
```

每个用户的 _custom_data_ 存储在一个单独的 JSON 对象中。

不要将敏感数据放入 _custom_data_

你可能会使用 Management API 获取包含 _custom_data_ 的用户资料，并将其发送到前端应用程序或外部后端服务。因此，将敏感信息放入 _custom_data_ 可能会导致数据泄露。

如果你仍然想将敏感信息放入 _custom_data_，我们建议先对其进行加密。仅在受信任的方（如你的后端服务）中进行加密/解密，避免在前端应用程序中进行。这些措施将最大限度地减少用户的 _custom_data_ 因意外泄露而造成的损失。

你可以使用 [Admin Console](/user-management/manage-users/#view-and-update-the-user-profile) 或 [Logto Management API](/user-management/manage-users/#manage-via-logto-management-api) 更新用户的 _custom_data_，例如 [`PATCH /api/users/:userId`](https://openapi.logto.io/operation/operation-patch-api-users-parameter)。

谨慎更新

更新用户的 _custom_data_ 将完全覆盖存储中的原始内容。

例如，如果你调用更新 _custom_data_ API 的输入如下（假设原始 _custom_data_ 是之前显示的示例数据）：

```json
{
  "customDataBaz": {
    "baz": "baz"
  }
}
```

那么更新后的 _custom_data_ 值应为：

```json
{
  "customDataBaz": {
    "baz": "baz"
  }
}
```

也就是说，更新后的字段值与之前的值无关。
