# システム制限

システム制限は、Logto の安定性と公平性を保つために設けられています。これらは課金とは別に管理されています。

**要点まとめ**

- **Dev テナント**：固定上限（ハードリミット）
- **Pro テナント**：初期上限（ソフトリミット）。正当な利用であれば引き上げ可能です。
- **Enterprise テナント**：契約で定義されたカスタマイズ可能な上限

料金ページでの **無制限** とは、_公正利用の範囲内で事前に設定された課金上限がない_ ことを意味します。プラットフォームの安定性を守るため、**初期ガードレール**（エンティティ上限やレートリミットなど）を適用していますが、正当なワークロードには引き上げが可能です。

## 制限の種類 \{#types-of-limits}

### システムリソースおよびエンティティ上限 \{#system-resource-and-entity-caps}

アプリケーション、ロール、組織、コネクター、Webhook などのオブジェクトや、アクセス トークン発行など一部の利用量ベースのシステムリソースに対する最大数です。

### レートリミット \{#rate-limit}

テナントごとに適用されるリクエストレベルの保護です。

デフォルト：**1 テナントあたり 10 秒間に 200 リクエスト**（トークンバケット方式）。これはおおよそ **20 RPS の持続的な処理** に相当し、バースト時にはバケット容量まで許容されます。

トラフィックが正当でこの上限を継続的に超える場合は、個別に引き上げを検討します。

## 上限に達した場合どうなるか \{#what-happens-when-you-hit-a-limit}

ガードレールが発動すると、リクエストは次のようなエラーで失敗する場合があります：

- **HTTP 429**（レートリミット）
- **HTTP 4xx**（エンティティ上限や機能ガードレールに関連するメッセージ付き）

これはデータを削除するものではなく、上限を超えた操作のみをブロックします。

## 上限引き上げのリクエスト方法 \{#request-a-limit-increase}

上限に達し、利用用途が正当な場合はご連絡ください。Pro および Enterprise では、通常すぐに上限を引き上げ可能です。

ご連絡時には以下をお知らせください：

- テナント ID（またはテナントドメイン）
- 達した上限（機能名＋必要な数値）
- （RPS リクエストの場合）想定トラフィック：平均 RPS、ピーク RPS、バーストパターン
- 影響を受けるエンドポイント（例：サインイン、トークン、JWKS、userinfo、Management API）
- 必要となる時期

[お問い合わせはこちら](https://logto.io/contact)

Dev テナントの上限は **ハードリミット** であり、開発ワークロード向けに設定されています。もし本番環境に近い環境が必要な場合は、[テナントを Pro へ変換](/logto-cloud/tenant-settings#tenant-types-dev-vs-prod) できます。

Dev テナントには別途 [データ保持ポリシー](/logto-cloud/dev-tenant-data-retention) も適用されます。
