# 實務上的角色型存取控制 (RBAC in practice)：為你的應用程式實作安全的授權 (Authorization)

你是否在為你的應用程式實作一個安全且可擴展的授權 (Authorization) 系統而苦惱？**角色型存取控制 (RBAC, Role-based Access Control)** 是管理使用者權限的業界標準，但正確實作卻充滿挑戰。本教學將以實際的內容管理系統（CMS）案例，帶你建立一套健全的 RBAC 系統。

跟著本指南，你將學會：

- ✨ 如何設計與實作細緻的權限，精確掌控操作
- 🔒 如何將權限有意義地組織成角色的最佳實踐
- 👤 有效處理資源擁有權的技巧
- 🚀 讓你的授權 (Authorization) 系統具備可擴展性與可維護性的方式
- 💡 以真實 CMS 案例實作

本教學的完整原始碼可在 [GitHub](https://github.com/logto-io/rbac-sample) 取得。

## 理解 RBAC 基礎 \{#understanding-rbac-fundamentals}

角色型存取控制 (RBAC) 不僅僅是將權限分配給使用者，更是建立一套結構化的授權 (Authorization) 方法，兼顧安全性與可維護性。

你可以在 Auth Wiki 進一步了解 [什麼是 RBAC](https://auth-wiki.logto.io/rbac)。

以下是我們在實作中會遵循的關鍵原則：

### 細緻的權限設計 \{#fine-grained-permission-design}

細緻的權限讓你能精確控制使用者在系統中的操作。與其僅用「管理員」或「一般使用者」這類寬泛的存取層級，我們會定義使用者可對資源執行的具體動作。例如：

- `read:articles` - 檢視系統中任何文章
- `create:articles` - 建立新文章
- `update:articles` - 修改現有文章
- `publish:articles` - 變更文章發佈狀態

### 資源擁有權與存取控制 \{#resource-ownership-and-access-control}

資源擁有權是我們 CMS 授權 (Authorization) 設計中的核心概念。RBAC 定義不同角色可執行的動作，而擁有權則為存取控制增添個人層面：

- 作者自動擁有他們創建的文章的存取權
- 這種自然的擁有權模型讓作者永遠可以檢視與編輯自己的內容
- 系統在處理文章操作時會同時檢查角色權限或擁有權
- 例如，即使沒有 `update:articles` 權限，作者仍可編輯自己的文章
- 這種設計減少了額外角色權限的需求，同時維持安全性

這種雙層設計（角色 + 擁有權）讓系統更直覺且安全。出版者與管理員仍可透過角色權限管理所有內容，而作者則保有對自己作品的控制權。

## 設計安全的 API \{#designing-a-secure-apis}

我們先從設計 CMS 核心功能的 API 端點開始：

```
GET    /api/articles         # 列出所有文章
GET    /api/articles/:id     # 取得特定文章
POST   /api/articles        # 建立新文章
PATCH  /api/articles/:id    # 更新文章
DELETE /api/articles/:id    # 刪除文章
PATCH  /api/articles/:id/published  # 變更發佈狀態
```

### 為 API 實作存取控制 \{#implement-access-control-for-your-api}

對每個端點，我們需考慮兩個存取控制面向：

1. 資源擁有權 —— 使用者是否擁有此資源？
2. 角色型權限 —— 使用者的角色是否允許此操作？

以下是每個端點的存取控制邏輯：

| 端點                              | 存取控制邏輯                                        |
| --------------------------------- | --------------------------------------------------- |
| GET /api/articles                 | - 具有 `list:articles` 權限者，或作者可見自己的文章 |
| GET /api/articles/:id             | - 具有 `read:articles` 權限者，或該文章作者         |
| POST /api/articles                | - 具有 `create:articles` 權限者                     |
| PATCH /api/articles/:id           | - 具有 `update:articles` 權限者，或該文章作者       |
| DELETE /api/articles/:id          | - 具有 `delete:articles` 權限者，或該文章作者       |
| PATCH /api/articles/:id/published | - 僅限具有 `publish:articles` 權限者                |

### 建立可擴展的權限系統 \{#creating-a-permission-system-that-scales}

根據 API 存取需求，我們可以定義下列權限：

| 權限             | 說明                   |
| ---------------- | ---------------------- |
| list:articles    | 檢視系統中所有文章列表 |
| read:articles    | 閱讀任何文章的完整內容 |
| create:articles  | 建立新文章             |
| update:articles  | 修改任何文章           |
| delete:articles  | 刪除任何文章           |
| publish:articles | 變更發佈狀態           |

注意，這些權限僅在存取非自己擁有的資源時才需要。文章擁有者可自動：

- 檢視自己的文章（不需 `read:articles`）
- 編輯自己的文章（不需 `update:articles`）
- 刪除自己的文章（不需 `delete:articles`）

## 建立有效的角色 \{#building-effective-roles}

現在我們已定義 API 與權限，可以依邏輯將這些權限分組成角色：

| 權限 / 角色      | 👑 管理員 (Admin)            | 📝 出版者 (Publisher)        | ✍️ 作者 (Author)     |
| ---------------- | ---------------------------- | ---------------------------- | -------------------- |
| **說明**         | 完整系統存取權，全面內容管理 | 可檢視所有文章並管理發佈狀態 | 可在系統中建立新文章 |
| list:articles    | ✅                           | ✅                           | ❌                   |
| read:articles    | ✅                           | ✅                           | ❌                   |
| create:articles  | ✅                           | ❌                           | ✅                   |
| update:articles  | ✅                           | ❌                           | ❌                   |
| delete:articles  | ✅                           | ❌                           | ❌                   |
| publish:articles | ✅                           | ✅                           | ❌                   |

**注意**：作者自動擁有對自己文章的讀取 / 更新 / 刪除權限，無論角色權限為何。

每個角色都針對特定職責設計：

- **管理員 (Admin)**：全面掌控 CMS，包括所有文章操作
- **出版者 (Publisher)**：專注於內容審核與發佈管理
- **作者 (Author)**：專注於內容創作

這樣的角色結構明確分工：

- 作者專注於內容創作
- 出版者管理內容品質與可見性
- 管理員維護整體系統

## 前端整合 Logto RBAC \{#integrate-your-frontend-with-logto-rbac}

現在我們已在 Logto 設定好 RBAC，可以開始整合到前端。

首先，依照 [Logto 快速入門](/quick-starts) 將 Logto 整合進你的應用程式。

本例以 [React](/quick-starts/react) 示範。

完成 Logto 設定後，需新增 RBAC 設定讓 Logto 能正確運作。

```tsx
// frontend/src/App.tsx

const logtoConfig: LogtoConfig = {
  appId: LOGTO_APP_ID, // 你在 Logto Console 建立的 app ID
  endpoint: LOGTO_ENDPOINT, // 你在 Logto Console 建立的 endpoint
  resources: [API_RESOURCE], // 你在 Logto Console 建立的 API resource identifier，例如 https://api.cms.com
  // 前端可能會向 API resource 請求的所有 scopes
  scopes: [
    'list:articles',
    'create:articles',
    'read:articles',
    'update:articles',
    'delete:articles',
    'publish:articles',
  ],
};
```

如果你已登入，記得登出並重新登入讓設定生效。

當使用者透過 Logto 登入並請求上述 API 資源的存取權杖 (Access token) 時，Logto 會根據使用者角色將相關 scopes（權限）加入存取權杖 (Access token)。

你可以透過 `useLogto` hook 的 `getAccessTokenClaims` 取得存取權杖 (Access token) 中的 scopes。

```tsx
// frontend/src/hooks/use-user-data.ts

import { useLogto } from '@logto/react';
import { API_RESOURCE } from '../config';
import { useState, useEffect } from 'react';

export const useUserData = () => {
  const { getAccessTokenClaims } = useLogto();
  const [userScopes, setUserScopes] = useState<string[]>([]);
  const [userId, setUserId] = useState<string>();

  useEffect(() => {
    const fetchScopes = async () => {
      const token = await getAccessTokenClaims(API_RESOURCE);
      setUserScopes(token?.scope?.split(' ') ?? []);
      setUserId(token?.sub);
    };

    fetchScopes();
  }, [getAccessTokenClaims]);

  return { userId, userScopes };
};
```

你可以利用 `userScopes` 判斷使用者是否有權限存取資源。

```tsx
// frontend/src/pages/Dashboard.tsx

const Dashboard = () => {
  const { userId, userScopes } = useUserData();
  // ...

  return (
    <div>
      {/* ... */}
      {(userScopes.includes('delete:articles') || article.ownerId === userId) && (
        <button
          onClick={() => handleDelete(article.id)}
          className="text-red-600 hover:text-red-900"
        >
          Delete
        </button>
      )}
    </div>
  );
};
```

## 後端整合 Logto RBAC \{#integrate-your-backend-with-logto-rbac}

接下來，將 Logto RBAC 整合進你的後端。

### 後端授權 (Authorization) 中介軟體 \{#backend-authorization-middleware}

首先，我們需要在後端新增一個中介軟體，檢查使用者權限、驗證是否已登入，以及是否有權存取特定 API。

```javascript
// backend/src/middleware/auth.js

const { createRemoteJWKSet, jwtVerify } = require('jose');

const getTokenFromHeader = (headers) => {
  const { authorization } = headers;
  const bearerTokenIdentifier = 'Bearer';

  if (!authorization) {
    throw new Error('Authorization header missing');
  }

  if (!authorization.startsWith(bearerTokenIdentifier)) {
    throw new Error('Authorization token type not supported');
  }

  return authorization.slice(bearerTokenIdentifier.length + 1);
};

const hasScopes = (tokenScopes, requiredScopes) => {
  if (!requiredScopes || requiredScopes.length === 0) {
    return true;
  }
  const scopeSet = new Set(tokenScopes);
  return requiredScopes.every((scope) => scopeSet.has(scope));
};

const verifyJwt = async (token) => {
  const JWKS = createRemoteJWKSet(new URL(process.env.LOGTO_JWKS_URL));

  const { payload } = await jwtVerify(token, JWKS, {
    issuer: process.env.LOGTO_ISSUER,
    audience: process.env.LOGTO_API_RESOURCE,
  });

  return payload;
};

const requireAuth = (requiredScopes = []) => {
  return async (req, res, next) => {
    try {
      // 取得權杖
      const token = getTokenFromHeader(req.headers);

      // 驗證權杖
      const payload = await verifyJwt(token);

      // 將使用者資訊加入請求
      req.user = {
        id: payload.sub,
        scopes: payload.scope?.split(' ') || [],
      };

      // 驗證所需 scopes
      if (!hasScopes(req.user.scopes, requiredScopes)) {
        throw new Error('Insufficient permissions');
      }

      next();
    } catch (error) {
      res.status(401).json({ error: 'Unauthorized' });
    }
  };
};

module.exports = {
  requireAuth,
  hasScopes,
};
```

如你所見，此中介軟體會驗證前端請求是否帶有有效的存取權杖 (Access token)，並檢查該權杖的受眾 (audience) 是否與我們在 Logto Console 建立的 API 資源一致。

驗證 API 資源的原因在於，我們的 API 資源實際上代表 CMS 後端的資源，所有 CMS 權限都與此 API 資源綁定。

由於這個 API 資源在 Logto 代表 CMS 資源，因此前端在呼叫後端 API 時，會帶上對應的存取權杖 (Access token)：

```typescript
// frontend/src/hooks/use-api.ts
export const useApi = () => {
  const { getAccessToken } = useLogto();

  return useMemo(
    () =>
      async (endpoint: string, options: RequestInit = {}) => {
        try {
          // 取得 API 資源的存取權杖 (Access token)
          const token = await getAccessToken(API_RESOURCE);

          if (!token) {
            throw new ApiRequestError('Failed to get access token');
          }

          const response = await fetch(`${API_BASE_URL}${endpoint}`, {
            ...options,
            headers: {
              'Content-Type': 'application/json',
              // 將存取權杖 (Access token) 加入請求標頭
              Authorization: `Bearer ${token}`,
              ...options.headers,
            },
          });

          // ... 處理回應

          return await response.json();
        } catch (error) {
          // ... 錯誤處理
        }
      },
    [getAccessToken]
  );
};
```

現在我們可以用 `requireAuth` 中介軟體保護 API 端點。

### 保護 API 端點 \{#protecting-api-endpoints}

對於僅允許特定權限使用者存取的 API，可直接在中介軟體中加上限制。例如，建立文章 API 僅允許擁有 `create:articles` 權限的使用者存取：

```javascript
// backend/src/routes/articles.js

const { requireAuth } = require('../middleware/auth');

router.post('/articles', requireAuth(['create:articles']), async (req, res) => {
  // ...
});
```

對於需同時檢查權限與資源擁有權的 API，可使用 `hasScopes` 函式。例如，文章列表 API，擁有 `list:articles` 權限者可存取所有文章，否則僅能存取自己創建的文章：

```javascript
// backend/src/routes/articles.js

const { requireAuth, hasScopes } = require('../middleware/auth');

router.get('/articles', requireAuth(), async (req, res) => {
  try {
    // 有 list:articles 權限則回傳所有文章
    if (hasScopes(req.user.scopes, ['list:articles'])) {
      const articles = await articleDB.list();
      return res.json(articles);
    }

    // 否則僅回傳使用者自己的文章
    const articles = await articleDB.listByOwner(req.user.id);
    res.json(articles);
  } catch (error) {
    res.status(500).json({ error: 'Failed to fetch articles' });
  }
});
```
