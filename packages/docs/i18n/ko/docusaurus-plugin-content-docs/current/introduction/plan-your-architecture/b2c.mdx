# B2C 서비스를 위한 단일 테넌트 아키텍처

## 아키텍처 \{#architecture}

소비자 애플리케이션에서는 사용자가 자신의 계정을 완전히 소유하고 아이덴티티와 인증을 제어하며, 기업이나 조직과 같은 다른 "중간 계층" 당사자의 개입이 없습니다. 이것이 B2C와 B2B 아이덴티티 아키텍처의 주요 차이점입니다.

## 요구 사항을 Logto의 지원 도구 키트에 연결하기 \{#connect-your-requirements-to-logtos-support-toolkit}

이 아키텍처에는 관리 시나리오에 관련된 두 개의 주요 당사자가 포함됩니다. 특정 요구 사항과 목표에 따라, 이들 당사자 모두 또는 일부만 관련될 수 있습니다.

우리는 일반적인 사용 사례를 요약하여 각 사용자 아이덴티티 관리 작업의 주요 목표와 관련 제품 및 API를 강조했습니다. 귀하의 요구 사항을 우리의 서비스에 매핑하여 빠르게 시작할 수 있습니다.

| 사용자             | 목표                                                                                    | Logto 제품 및 API                                                                                                                                                |
| ------------------ | --------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 개발자             | 사용자 아이덴티티 시스템을 관리하고 보호하며 아이덴티티 데이터베이스와 직접 작업합니다. | <ul><li>[Logto Console](https://cloud.logto.io)</li><li>[Logto Management API](https://openapi.logto.io/)</li></ul>                                              |
| 최종 사용자/소비자 | [자신의 인증 및 개인 정보를 관리합니다.](/end-user-flows)                               | <ul><li>[Logto Management API](https://openapi.logto.io/)</li><li>[Account API](https://openapi.logto.io/operation/operation-getaccountcentersettings)</li></ul> |
